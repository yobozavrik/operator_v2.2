# Production Rules

## Module Role
Production — це контур начальника виробництва та операційного виконання зміни.

Його головне призначення:
- показувати, що треба запускати у зміну
- давати зрозумілий порядок пріоритетів
- підсвічувати блокери
- показувати стан виконання
- допомагати закривати план зміни без зайвого аналітичного шуму

## Main Product Principle
Production не повинен бути просто "ще одним красивим дашбордом".

Це робочий інструмент для:
- щоденного управління зміною
- контролю черги
- контролю дефіцитів та блокерів
- розуміння, що запускати зараз, що далі, а що заблоковано

## Core Users
Основні користувачі production-контуру:
- Начальник виробництва
- Операційний директор

## What Production Must Answer
Правильний production-екран повинен швидко відповідати на питання:
- що запускати зараз
- що пріоритетне на цю зміну
- що блокує виконання
- що вже зроблено
- що ризикує не встигнути
- що треба закрити до кінця зміни

## What Must Be Preserved
### 1. Operational usefulness
Екран має залишатися практичним і робочим, а не перетворюватися на абстрактну аналітику.

### 2. Fast decision-making
Начальник виробництва має бачити пріоритети за секунди, а не витрачати час на розшифровку інтерфейсу.

### 3. Dense but readable workflow
Щільність даних допустима, якщо вона допомагає роботі.
Не треба штучно "спрощувати" Production до втрати змісту.

## What Must Be Improved
### 1. Shift-first logic
Інтерфейс має бути побудований навколо зміни, а не навколо абстрактного набору метрик.

### 2. Clear priority order
Черга має мати зрозумілий порядок:
- критичне
- термінове
- звичайне
- резерв

### 3. Blocker visibility
Блокери мають бути очевидними:
- матеріали
- персонал
- обладнання
- логістичні залежності
- попередні етапи

### 4. Execution states
Кожна задача або партія має мати чіткий стан, який швидко читається.

## Recommended Decision Flow
Production має вести користувача по ланцюжку:
1. пріоритет зміни
2. що запускати зараз
3. що заблоковано
4. що вже в роботі
5. що виконано
6. що під ризиком до кінця зміни

## Visual Rules
### Allowed
- щільний робочий інтерфейс
- чіткі статуси
- виразна черга
- сильні сигнали для блокерів і критичних задач

### Not allowed
- декоративні ефекти, які заважають роботі
- надлишкова "вітринність"
- змішування аналітики і дій без ієрархії
- неочевидні переходи між пріоритетом, задачею і фактичним виконанням

## Terminology Rules for Production
Використовувати стабільно:
- **зміна**
- **черга**
- **пріоритет**
- **блокер**
- **у роботі**
- **виконано**
- **затримка**
- **ризик**
- **до запуску**
- **до випуску**
- **готовність**
- **відвантаження**
- **персонал**
- **виробництво**

## Red Lines
- Не перетворювати Production на abstract analytics view.
- Не ховати пріоритети за декоративними шарами.
- Не змішувати мову інтерфейсу: production-UI має бути лише українською.
- Не розривати зв’язок між чергою, задачею, станом і дією.

## Current Focus
Найближча задача для production-контуру:
- зібрати його як справжній shift execution layer
- привести мову до чистої української
- зробити пріоритети, блокери і готовність максимально читабельними
- зберегти щільність там, де вона реально допомагає роботі
