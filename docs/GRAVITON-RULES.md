# Graviton Rules

## Module Role
Graviton — один із ключових операційних модулів ERP.

Це не просто аналітичний екран і не просто вітрина метрик.
Його основне призначення:
- швидко показувати проблемні зони мережі
- допомагати бачити дефіцити та пріоритети
- вести користувача від сигналу до дії
- дозволяти швидко формувати виробничу заявку

## Main Product Principle
Graviton не треба "перемальовувати з нуля", якщо поточний щільний картковий патерн уже сильний.

Правильний підхід:
- зберігати сильну основу
- прибирати хаос
- посилювати зміст
- уніфікувати терміни
- робити інтерфейс точнішим, а не стерильнішим

## What Must Be Preserved
### 1. Dense operational overview
Щільна карткова сітка та швидкий огляд — це сильна сторона Graviton.

Її не треба ламати лише заради "чистішого дизайну".

### 2. Fast scanning
Користувач має швидко зчитувати:
- де критично
- де великий обсяг
- де проблема по конкретній категорії
- куди натискати далі

### 3. Operational rhythm
Graviton має залишатися робочим інструментом для живого щоденного управління, а не перетворюватися на повільний executive-only dashboard.

## What Must Be Improved
### 1. Language consistency
- Увесь UI Graviton має бути українською.
- Не змішувати українську, російську й англійську.
- Усі кнопки, модалки, картки, підказки, статуси й заголовки мають говорити однією мовою.

### 2. Semantic consistency
Картки, модалки й деталізація мають використовувати однакову логіку:
- сигнал
- причина
- дія
- підтвердження

### 3. Sorting and priority
Порядок елементів має підсилювати рішення:
- критичне — вище
- великий обсяг — вище
- менш важливе — нижче

### 4. Click logic
Клік по картці або категорії має бути очевидним:
- відкриває деталі
- показує, що саме потрібно обрати
- дає перейти до формування заявки

### 5. Order flow clarity
Відбір, модалка категорії, store-specific flow і підтвердження заявки мають працювати як один сценарій.

## Required User Journey
Для Graviton правильний ланцюжок такий:
1. побачити сигнал
2. зрозуміти, яка категорія/позиція критична
3. відкрити деталі
4. обрати позиції
5. сформувати заявку
6. перевірити заявку
7. надіслати

## Visual Rules
### Allowed
- щільна сітка карток
- сильні акценти для критичних станів
- виразна ієрархія ваги / кг / критичності
- компактні статуси

### Not allowed
- декоративні ефекти, що заважають читанню
- кольори без смислового навантаження
- складні написи там, де має бути короткий сигнал
- розмиття термінів між карткою, модалкою і заявкою

## Terminology Rules for Graviton
У Graviton особливо важливо не плодити синоніми для однієї і тієї ж сутності.

Використовувати стабільно:
- **дефіцит**
- **залишок**
- **фактичний залишок**
- **треба виготовити**
- **категорія**
- **позиція**
- **обрати**
- **заявка**
- **критично**
- **ризик**
- **стабільно**
- **оновити залишки**
- **розподіл**

## Red Lines
- Не спрощувати Graviton до рівня "красивої панелі", якщо це вбиває операційну користь.
- Не розривати логіку між карткою, категорією, товаром і заявкою.
- Не вводити нові терміни без перевірки проти `UI-TERMINOLOGY.md`.
- Не ламати щільну карткову сітку, якщо немає дуже сильної продуктової причини.

## Current Focus
Найближча задача по Graviton:
- завершити мовну чистку
- вирівняти підписи та стани
- довести модалки й дії до єдиного сценарію
- зберегти сильну карткову основу
