# ERP Product Rules

## Project
`D:\operator-main`

## Product Goal
Перетворити ERP / Production Hub на сильну, цілісну, візуально зрозумілу й управлінськи правильну систему.

Головний фокус не на тому, щоб зробити "красивіше", а на тому, щоб зробити:
- правильніше
- ефективніше
- системніше
- зрозуміліше для ухвалення рішень

## Core Roles
Система будується під 3 ключові ролі:
- Власник
- Операційний директор
- Начальник виробництва

## Language Rules
### External communication
- З Дімою спілкування ведеться **строго російською мовою**.

### ERP interface
- Увесь інтерфейс ERP має бути **строго українською мовою**.
- Не змішувати в UI:
  - російську
  - українську
  - англійську
- Якщо англійський термін не є брендом або усталеним технічним словом, він має бути перекладений.

## UX / Product Rules
- Не ламати сильні робочі патерни заради редизайну.
- Особливо: **не ламати карткову сітку Graviton**, якщо вона вже добре працює як інструмент огляду.
- Якщо патерн сильний, його треба:
  - зберегти
  - очистити
  - зробити точнішим
  - зробити консистентним
- Редизайн має посилювати:
  - семантику
  - порядок
  - пріоритетність
  - дії
  - зрозумілість

## Information Architecture Direction
Верхній рівень системи будується навколо ролей:
- `/`
- `/owner`
- `/ops`
- `/production-chief`
- `/workshops`

Поточні модулі та цехові екрани залишаються як рівні деталізації.

## Decision Flow Principle
Кожен сильний екран має вести по ланцюжку:
1. сигнал
2. причина
3. дія
4. деталізація

## UI Semantics
- Кольори мають позначати стан, а не лише декор.
- Важливі KPI та картки мають бути швидко сканованими.
- Інтерфейс має залишатися щільним там, де це допомагає роботі.
- Не перетворювати ERP на стерильний SaaS, якщо це вбиває операційну корисність.

## Graviton-Specific Rules
- Graviton є одним із ключових модулів системи.
- Його не треба "перемальовувати з нуля", якщо поточна щільна карткова сітка вже сильна.
- Треба покращувати:
  - підписи
  - стани
  - логіку сортування
  - клік-сценарії
  - модалки
  - узгодженість термінів

## Work Style
- Працювати завершеними блоками.
- Не повертатися з порожніми повідомленнями типу "продовжую".
- Після кожного блоку фіксувати:
  - які файли змінено
  - що зроблено
  - що стало краще
  - що далі
